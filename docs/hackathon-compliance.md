## Human-in-the-Loop Workflow

```
1. Agent generates plan (read-only operation)
2. Coordinator reviews each mission plan in detail:
   - Emergency location, type, urgency, people affected
   - Assigned volunteer (name, skills, location)
   - Allocated resource (type, quantity, location)
   - Full structured reasoning (priority, volunteer match, resource, risk, next action)
3. Coordinator selects/deselects individual plans
4. Coordinator clicks "Approve and Create Missions" (explicit action)
5. Only then does the agent write to MongoDB
6. Missions become active, statuses update, audit log records the action
```

This satisfies the "human oversight" requirement — no autonomous writes occur without coordinator approval.

## Reliability & Demo Safety

| Feature | Behavior |
|---|---|
| Gemini succeeds | Plans generated by Gemini API with full reasoning |
| Gemini quota/billing error | Yellow banner shown: "Gemini unavailable — fallback planner active." Plans still generated by deterministic planner. Demo never breaks. |
| Gemini not configured | Silent fallback to deterministic planner (no error shown, just engine label changes) |
| MongoDB disconnected | API returns 500 with clear error message |

The fallback ensures the demo is always functional regardless of API quota, billing, or network issues.

## Agent Reasoning Transparency

For every mission plan, the agent shows:

1. **Priority Reason** — Why this request is at this urgency level
2. **Volunteer Match Reason** — Why this specific volunteer was chosen (skill score, location proximity)
3. **Resource Allocation Reason** — Why this resource type matches the emergency
4. **Risk Level** — Critical / High / Medium / Low with color coding
5. **Next Action** — Concrete immediate step for the field team

This level of transparency demonstrates true agentic reasoning, not just text generation.
